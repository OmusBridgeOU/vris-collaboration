# layers/vris 新規イベントページ 要件定義書

## 1. 文書の目的

本書は `layers/vris` における最新イベント実装である `2026EarlySpring` を基準として、次回以降の新規イベントページを開発するための要件を定義する。  
単なる現状整理ではなく、「何を流用し、何を差し替え、どこを改善対象とするか」を明確にすることを目的とする。

## 2. 要件定義の前提

### 2.1 基準実装

新規イベントページ開発時の基準は、現在 `/` で表示されている以下の実装とする。

- `app/pages/index.vue`
- `app/components/ht/Ht2026EarlyspringTop.vue`
- `app/components/ho/sections/2026Earlyspring/*.vue`
- `app/components/ho/HoTheHeader.vue`
- `app/components/ho/HoTheFooter.vue`
- `app/layouts/2025WinterTeaser.vue`
- `content/docs/2026early-spring/*.md`

### 2.2 事前確認済み事項

本書作成にあたり、以下を確認した。

- 2026 系トップページの全セクション実装
- 2026 系で未採用だが存在する補助セクション実装
  - `Ho2026EarlyspringEventInfoSection.vue`
  - `Ho2026EarlyspringExhibitorsSection.vue`
  - `Ho2026EarlyspringLightningTalkSection.vue`
  - `Ho2026EarlyspringFaqSection.vue`
- 共有レイアウト、共有ヘッダー、共有フッター、Nuxt Content 取得処理
- 2026 Early Spring 関連の Markdown 文書本文
- runtimeConfig / nuxt.config / GTM / structured data の現行状態

## 3. 新規開発の目的

- `2026EarlySpring` と同等以上の情報量と導線品質を持つ新しいイベントページを追加開発する
- 2026 実装で確立されたページ構成、セクション分割、ビジュアルトーン、外部導線、文書導線を継承する
- 2026 実装に残っているイベント固有ハードコードを整理し、新規イベントへ安全に差し替えられる状態にする

## 4. 対象システム

- 対象ディレクトリ: `layers/vris`
- 実装基盤: Nuxt 3
- 配信形態: CSR (`ssr: false`)
- 配備想定: Cloudflare Pages
- 多言語: 日本語 / 英語

## 5. 新規イベントページの完成イメージ

- 2026 Early Spring と同様に、1ページでイベント全体を把握できる LP とする
- 各種関連文書、問い合わせ、SNS、チケット購入への導線を同一サイト内で完結させる
- イベント固有の世界観を画像・コピー・配色で表現しつつ、情報設計は 2026 を踏襲する
- 将来イベントに再利用できるよう、イベント固有情報は局所化する

## 6. 機能要件

### 6.1 ページ構成要件

新規イベントページは、少なくとも以下の構成を持つこと。

1. Hero
2. Ticket
3. About / Intro
4. Schedule
5. Highlights / Contents
6. Access
7. Documents
8. Corporate
9. Staff
10. Archive or Related

2026 実装で現在採用されている追加構成は、イベント内容に応じて継承または再編できること。

- Circle
- Art
- LT Speakers

2026 実装で存在はするがトップで未採用の補助構成は、必要に応じて採用判断できること。

- EventInfo
- Exhibitors
- LightningTalk summary
- FAQ excerpt

### 6.2 Hero 要件

- イベントのメインコピーを `h1` で表示すること
- 開催日時、会場、補足情報を表示すること
- 主 CTA を 1 件以上表示すること
- イベントロゴまたはキービジュアルを表示すること
- CTA は外部リンク時に `target="_blank"` と `rel="noopener"` を設定すること

### 6.3 Ticket 要件

- 参加費または券種情報を表示すること
- 購入期限、販売条件、補足注意事項を表示できること
- チケット購入導線を設置すること

### 6.4 About / Intro 要件

- イベント趣旨、テーマ、特徴を複数の観点で説明すること
- 2026 実装では 3 カード構成であるため、新規イベントでも複数カードまたは同等の情報ブロックで説明すること

### 6.5 Schedule 要件

- 当日の主要時刻と内容を時系列で表示すること
- 補足注意事項を表示できること
- タイムテーブル画像を併記できること

### 6.6 Highlights / Contents 要件

- 主要企画を複数カードで紹介すること
- 画像カルーセルを持てること
- カルーセル画像をダイアログで拡大表示できること
- 2026 実装と同様に、コンテンツ紹介カードとビジュアルカルーセルを同居できること

### 6.7 Circle / Exhibitor 要件

- 出展サークルまたは出展者を一覧表示できること
- 画像がある場合はスライダーで見せられること
- 追加画像がある場合はダイアログ内で縦並び表示できること
- 画像未確定時は代替画像を設定できること

### 6.8 LT 要件

- 登壇者一覧を表示できること
- テーマまたは登壇タイトルを表示できること
- 外部プロフィールリンクを保持できること

### 6.9 Access 要件

- 会場名、会場説明、アクセス要点を表示すること
- 会場公式サイトへの導線を表示すること
- Google Maps 埋め込み等で場所を可視化すること

### 6.10 Documents 要件

- 主要ドキュメントへのリンク一覧を表示すること
- 少なくとも以下の文書群をリンク可能であること
  - 出展ガイドライン
  - 出展規約
  - LT ガイドライン
  - FAQ
  - イベントガイド
  - 問い合わせ

### 6.11 Corporate 要件

- 協力企業・団体・関連イベント等をロゴ付きで表示できること
- ロゴ、名称、役割、リンク先を持てること
- 外部リンクカードとして表示できること

### 6.12 Staff 要件

- スタッフ、デザイン、広報、スペシャルサンクス等をグルーピングして表示できること
- 役割名、表示名、任意の外部リンクを持てること
- グループ単位で表示・非表示を切り替えられること

### 6.13 Archive / Related 要件

- 前回イベント実績または関連イベント実績を掲載できること
- 来場者数、開催日、会場などの KPI を表示できること
- 参考出典リンクを表示できること

### 6.14 共通ヘッダー要件

- ロゴ表示
- 言語切替
- モバイルメニュー
- ドキュメント導線
- 外部導線
- イベントガイドバー

を持つこと。

ヘッダーは新規イベントに合わせて以下を差し替えること。

- docs へのリンク先 slug
- ガイドバー文言
- ガイドバー遷移先
- 必要に応じて旧イベント導線

### 6.15 共通フッター要件

- お問い合わせ
- 法人問い合わせ
- 広報問い合わせ
- 公式 SNS
- プレスリリース
- プライバシーポリシー
- コピーライト

を表示できること。

### 6.16 ドキュメントページ要件

- `content/docs/<event-slug>/*.md` を `docs/[...slug]` で表示できること
- Markdown の見出し、表、画像、箇条書き、コードブロックを適切にレンダリングできること
- 404 相当の扱いができること

### 6.17 多言語要件

- 日本語 / 英語を切り替えられること
- 少なくともトップページの主要 UI テキストは 2 言語対応すること
- 文書ページは個別文書の運用方針に従うが、ヘッダー・導線・共通 UI は切り替わること

### 6.18 計測・SEO 要件

- GTM を利用できること
- title / description / OGP / Twitter Card を設定できること
- 構造化データをイベント内容に合わせて差し替えられること

## 7. データ要件

### 7.1 イベント固有データ

新規イベントページで差し替え対象となるデータは少なくとも以下。

- イベント名称
- テーマ / コピー
- 開催日時
- 会場情報
- チケット URL / 料金
- 主要企画一覧
- スケジュール
- 出展者 / サークル / 登壇者 / スタッフ
- 協力企業
- ドキュメント slug
- 画像一式
- OGP / structured data

### 7.2 文書データ

- 格納形式: Markdown
- 格納場所: `content/docs/<event-slug>/*.md`
- 更新対象:
  - event notification
  - faq
  - exhibition guideline
  - exhibition terms
  - lt guideline
  - guide-circle-exhibition
  - guide-company-exhibition

### 7.3 画像データ

- 格納場所: `public/images/<EventKey>/`
- ページ内画像はイベント単位のディレクトリで管理すること
- 画像未確定時は `noimage16-9.png` などの共通代替画像を使用できること

## 8. 非機能要件

### 8.1 保守性

- セクションごとに Vue コンポーネントを分離すること
- イベント固有文言やリンクを特定コンポーネント内に閉じ込めるか、共通設定へ抽出すること
- 新規イベント追加時に既存イベントを壊さないこと

### 8.2 拡張性

- 2026 実装で未採用の補助セクションを再利用可能にすること
- セクションの追加、削除、並び替えが容易であること

### 8.3 品質

- `lint`, `typecheck`, `test:ut` を通せること
- 画像パス、外部リンク、文書リンクの不整合を残さないこと

### 8.4 レスポンシブ

- PC / SP で破綻なく表示できること
- カルーセル、カード、ヘッダー、CTA がタップ可能サイズを満たすこと

## 9. 新規開発時の継承ルール

### 9.1 必ず継承するもの

- 1ページ完結型の情報設計
- セクション分割構成
- 共通ヘッダー / フッターの考え方
- ドキュメントページ連携
- 画像カルーセル + ダイアログの UI パターン
- 日本語 / 英語の切り替え構造

### 9.2 必ず差し替えるもの

- 全イベント名
- 年号、開催日、会場
- slug
- 画像パス
- チケット / 問い合わせ / 外部リンク
- OGP / メタ / structured data
- ヘッダーの guide bar 文言とリンク
- フッターコピーライト

### 9.3 必ず見直すもの

- 使わないセクションの削除
- 採用するセクション順
- 2026 固有表現の残存
- docs のリンク構成
- 旧イベント導線の扱い

## 10. 現行 2026 実装から判明している制約・課題

新規イベントページ開発時には、以下をそのまま持ち込まないこと。

- `structured-data.client.ts` が 2025 Summer 固定内容のまま
- `HoTheHeader.vue` の docs リンクが `2026early-spring` 固定
- `HoTheFooter.vue` のコピーライトが 2025 固定
- `nuxt.config.ts` のグローバル meta が 2026 固定
- `app/pages/index.vue` が `2025-winter-teaser` レイアウト名を使用している
- `useSectionState.ts` の section 定義が 2025 系 10 セクション固定
- `cards` collection を取得しているが、2026 トップでは実質未使用
- `Ho2026EarlyspringHighlightsSection.vue` が存在しない `/images/2026EarlySpring/contents/utage.jpeg` を参照しており、実ファイルは `utage.png`
- `Ho2026EarlyspringAccessSection.vue` の英語文言で徒歩時間が `5-minute walk` になっており、日本語 `徒歩約2分` と不一致

## 11. 受入条件

以下を満たしたとき、新規イベントページ要件を満たしたと判定する。

- 新規イベント用トップページが追加され、主要情報が 1 ページで確認できる
- 新規イベント用 docs 一式が表示できる
- ヘッダー、フッター、guide bar、documents セクションが新イベント向けに更新されている
- Hero / Ticket / Schedule / Access / Documents / Corporate / Staff が正しく差し替わっている
- 画像パス切れ、リンク切れ、イベント名の旧残りがない
- メタ情報と structured data が新イベント用に更新されている
- lint / typecheck / 必要テストが通っている

## 12. 作成後セルフレビュー結果

本書作成時点で、以下を自己点検し反映済み。

- 2026 最新トップで実際に採用されているセクションのみを「現行必須」に分類した
- 2026 ディレクトリ内でも未採用の補助セクションは「任意採用」に分離した
- 2026 関連 Markdown 文書に存在するドキュメント群だけを文書要件に記載した
- 共有実装に残る年号固定、slug 固定、構造化データ固定を課題として明示した
- 実装とアセットの照合により、存在しない画像参照と日英文言不一致を課題として明示した
